## 2. 제품 비전과 플레이어 약속

비전:

> 작은 쥐구멍 하나에서 시작해 세상 모든 쥐의 문명을 만든다.

플레이어에게 주는 네 가지 약속:

1. 쥐들이 계속 움직여 화면을 바라보는 것만으로도 생동감이 있다.
2. 짧은 시간 안에 다음 목표가 보이고, 달성 순간 분명한 보상을 받는다.
3. 숫자가 커질 때 배경, 쥐의 도구, 쥐구멍도 함께 변한다.
4. 장기적으로 플레이어만의 쥐 문명과 역사가 남는다.

숫자만 커지는 방치형 게임이 아니라, 작은 생활 공간이 사회와 문명으로 변하는
과정을 관찰하는 게임을 지향합니다.

## 3. 전체 플레이 흐름

```text
쥐 출발
  → 인간 세계의 자원 지점으로 이동
  → 자원 획득
  → 쥐구멍 귀환과 저장
  → 즉시 강화
  → 시각 변화와 중간 보상
  → 새 지역 발견
  → 건설·직업·연구
  → 지하 마을과 문명 확장
```

현재 V0.4 Alpha는 `출발 → 획득 → 귀환 → 강화 → 지역 해금 → 재방문 →
지역 사건 → 현장 직접 행동 → 보육실 → 성체 합류 → 역할 배치 → 첫 영웅
선택 → 영웅 유대 임무 → Web 저장 복구 → 살아있는 쥐 행동`까지 구현되어
있습니다. V0.2 iPad 실기기 확인과 V0.4.5 자동 검증을
완료했습니다.

## 4. 성장과 보상의 기준 구조

성장은 서로 다른 네 축으로 보이게 합니다.

| 성장축 | 수치 효과 | 시각적 보상 |
| --- | --- | --- |
| 속도 | 이동속도 증가 | 후속 전용 스프라이트로 목도리·운동화 표현 예정 |
| 운반 | 한 번에 가져오는 양 증가 | 현재 운반 스프라이트, 후속 전용 가방 표현 예정 |
| 인구 | 동시에 일하는 쥐 증가 | 실제 이동 쥐 증가 |
| 쥐구멍 | 확장 레벨과 장기 시설 기반 | 후속 배경·전용 아트로 단계 표현 예정 |

보상은 네 시간 단위로 배치합니다.

| 보상 단위 | 목표 체감 | 예시 |
| --- | --- | --- |
| 즉시 | 수 초 | 치즈 입고 숫자, 소리, 작은 효과 |
| 업그레이드 | 1~3분 | 능력 증가와 장비 외형 변화 |
| 중간 이정표 | 3~8분 | 배경 소품, 주민, 저장 공간 추가 |
| 지역 해금 | 10~20분 | 새 배경, 자원, 위험, 이야기와 생산 보너스 |

다음 보상 UI는 이름, 진행도, 보상 미리보기를 함께 보여줍니다.

```text
다음 보상: 동네 편의점
누적 치즈 342 / 500

네온 편의점 배경 · 배낭 운반쥐
야간 폐기 사건 · 쥐구멍 저장실
```

## 5. 초기 지역 콘텐츠 계획

수치 기준은 `data/stages/stages.json`과 일치해야 합니다.

| 지역 | 누적 치즈 | 보너스 | 배경·자원 | 위험·사건 | 성장 판타지 |
| --- | ---: | ---: | --- | --- | --- |
| 낡은 부엌 | 0 | 1.0배 | 갈라진 벽, 치즈 부스러기 | 고양이, 냉장고 문 | 혼자 살아남기 |
| 식료품 창고 | 100 | 1.15배 | 선반, 곡물 자루 | 쥐덫, 대형 식량 | 저장과 동료 모집 |
| 동네 편의점 | 500 | 1.35배 | 네온, 냉장고, 과자 | 인간, 야간 폐기 | 인간 세계 첫 원정 |
| 대형 식당 | 2,000 | 1.65배 | 조리대, 화로, 고급 치즈 | 화재, 빵 축제 | 전문 직업의 시작 |
| 치즈 공장 | 10,000 | 2.0배 | 컨베이어, 배관, 숙성 치즈 | 기계, 황금치즈 | 자동화 문명의 씨앗 |

지역 변경은 색상만 바꾸는 것으로 완료 처리하지 않습니다. V0.2 이후 지역 완료
기준은 최소한 `고유 배경 실루엣 + 고유 자원 표현 + 해금 연출`입니다.

| 막 | 이야기 | 게임 시스템 |

## 7. 런타임 책임 지도

```text
Mouse / WorldView
  → 이동, 경로, 화면 표현
  → GameManager에 자원 입고 요청

GameManager
  → 자원, 강화, 지역, 통계의 단일 런타임 상태
  → EventBus로 변경 알림

GameUI / WorldView
  → EventBus를 구독해 화면만 갱신

SaveManager
  → JSON 저장, 백업, 검증, 스키마 마이그레이션

data/*.json
  → 스테이지와 후속 콘텐츠 정의
```

기술 세부 규칙은 `docs/technical_design.md`를 따릅니다.

## 8. 현재 우선순위

1. V0.4 영웅 선택이 iPad에서 오류와 화면 잘림 없이 실행될 것
2. 영웅의 이름·이야기·영구 효과가 확정 전에 명확히 보일 것
3. 다음 보상 목표가 항상 보일 것
4. 군락 전체 인구와 실제 채집 인구를 구분할 것
5. 영웅 검증 뒤에만 유대, 직업 레벨과 건물을 확장할 것

현재 단계에서는 시스템 수를 늘리는 것보다 기존 성장 루프의 보상 전달력을
높이는 것이 더 중요합니다.

## 9. 릴리스 품질 게이트

모든 버전은 다음을 통과해야 완료로 봅니다.

- Godot headless 테스트 통과
- GDScript 파싱 오류 없음
- 새 저장 필드가 있으면 `schema_version`과 마이그레이션 갱신
- 기존 저장과 손상 저장 복구 확인
- 1280×720 데스크톱과 대표 iPad 가로·세로 화면에서 UI 잘림 없음
- Safari 새 실행, 백그라운드 복귀, 홈 화면 PWA 재진입 확인
- 외부 이미지가 없어도 기본 도형으로 핵심 기능 실행 가능
- 문서의 구현 상태와 실제 코드·데이터 일치
- 의미 있는 변경은 changelog에 이유와 검증 기록

  부록에서 관리. 예산이 6시간→24시간(최대 4배)으로 늘었다고 해서 장기 시대가
  스코프로 복귀하는 것은 아닙니다.

막(幕)별 목표 예산(제안값, 추정 전까지는 확정 아님 — 정밀 실측은 불필요):

| 막 | 목표 예산(제안) | 근거 |
| --- | ---: | --- |
| 1막 생존(부엌) | 0.5–1h | V0.1, 이미 구현됨 |
| 2막 무리(창고) | 1.5–2.5h | V0.2–V0.4 메타시스템 대부분 이 구간 |
| 3막 원정(편의점) | 2–4h | V0.3, 이미 구현됨 |
| 4막 사회(식당+공장 시스템) | 9–12h | V0.5–V0.7, 대부분 미구현 — 최대 미확정 구간 |
| 5막 문명(지하 세력) | 6–8h | V0.8–V1.0 |
| 버퍼(측정오차 흡수) | 0.5–1h | 명시적 여유선, 암묵적으로 어느 막에 흡수시키지 않음 |

주의사항:

- **지역 ≠ 막**: 식당·공장은 이미 지역(배경·데이터)으로 구현돼 있지만 4막의
  "시스템"(직업·건물·연구)은 미구현입니다. 지역 존재를 4막 예산 소진으로
  착각하지 않습니다.
- V0.1–V0.4.5는 아직 시간 추정치가 채워지지 않았습니다. 위 표는 사전
  목표 상한선이며 검증된 배분이 아닙니다. 추정치는 `docs/balancing.md`의
  시간예산 원장에 누적합니다.

## 11. 시간예산 관리 원칙

향후 모든 버전 계획은 예산 없이 시스템을 추가하고 검증 없이 다음 버전으로
넘어가는 기존 패턴을 반복하지 않기 위해 다음을 따릅니다. `docs/plans/vX.Y.Z_*.md`
문서는 "완료" 상태로 표기되기 전 아래를 만족해야 합니다.

1. 목표 시간예산(범위)과 소속 막을 문서 상단에 선언했는가
2. 해결하는 결정 대기 보드 항목을 인용했는가(없으면 "해당 없음" 명시)
3. 기존 컴포넌트(미니게임 등)로 재사용 가능한 부분을 새로 만들기 전에 검토했는가
4. 완료 조건에 "`balancing.md` 원장에 추정 근거 기록"이 포함됐는가
5. 장기 시대(6막) 내용이 아님을 확인했는가
6. 추정이 소속 막 예산을 초과하면 컷·병합 우선순위가 다음 버전 계획에 반영됐는가

이 체크리스트는 **V0.5부터 적용**하며, V0.1–V0.4.5의 기존 계획 문서에는 소급
적용하지 않습니다.

원칙을 확정합니다. 원칙만 선언하고 끝내지 않도록 추적이 필요한 항목은 §13
결정 대기 보드에 등록합니다.

| 원칙 | 내용 | 추적 |
| --- | --- | --- |
| 연계 원칙 | 수집 이동→미니게임(현장행동/영웅유대)→지역 이벤트→성장 보상이 하나의 흐름으로 느껴져야 하며, 조작 방식·보상 언어가 시스템마다 단절되면 안 됨 | 미니게임 통합 리팩터링(`SequenceActionView`, `docs/plans/scope_review_1day_target.md` 3.3)의 근거 |

### 12.1 능동 참여 비중 확대 — 판단 기준

"참여를 늘린다"로 막연히 남기지 않고, 두 축으로 명시합니다.

- **① 유도 방법**: 어떤 계기·타이밍에 참여를 요청할 것인가(예: 다음 보상 카드에서
  자연스럽게 제시 vs 알림성 유도). 기존 next-reward 카드(`docs/game_design.md` §5)·
  미니게임 쿨다운 구조와 어떻게 연결할지 함께 정의합니다.
- **② 참여의 방향성**: 참여했을 때 실제로 무엇을 하게 되는가를 유형으로 분류합니다
  — 의사결정형(지역 이벤트 선택지 확장) / 숙련형(현재 점 미니게임처럼 조작 정확도
  요구) / 육성·꾸미기형(쥐 개체 개성·역할 커스터마이징과 결합).

두 축이 채워지기 전까지는 "장기" 상태로 두고, V0.5 계획 단계에서 연계 원칙·
개체 독립성 원칙과 함께 묶어 판단합니다.

### 12.2 이동 목표지점 정확도 — 분기 처리

이것이 실제로 관찰된 버그(`mouse.gd`/`world_view.gd` 경로 계산이 목표 지점에서
어긋남)라면 `docs/roadmap.md`의 작업 우선순위 규칙 1순위 "실행 오류 수정"에
해당해 스코프 재정렬보다 먼저 처리합니다. 반대로 향후 시스템에 적용할 일반
정밀도 기준이라면
`docs/technical_design.md` §8 검증 기준에 품질 게이트로 추가합니다. 실제 버그
여부는 확인이 필요하므로, 실행 시점에 증상 유무로 분기합니다.

## 14. 확정 원칙

- 주인공은 숫자가 아니라 실제로 움직이고 생활하는 쥐들입니다.
- 강화 버튼을 눌렀다면 수치뿐 아니라 화면에서도 변화가 보여야 합니다.
- 위험은 무작위 감점보다 준비하거나 선택할 수 있는 사건으로 만듭니다.
- 새 시스템은 기존 자원 원장, 이벤트 버스와 저장 구조를 우회하지 않습니다.
- 모바일에서 읽기 어렵거나 누르기 어려운 UI는 완료된 기능으로 보지 않습니다.
- V1 이전에는 로컬 단일 플레이 완성도를 온라인 기능보다 우선합니다.
- 새 시스템은 소속 막의 시간예산을 초과한 상태로 착수하지 않습니다.
