# V0.4.4 작업 계획 — Web 재접속 저장 복구

작성일: 2026-07-19
상태: 구현·검증·운영 배포 완료

## 1. 문제

Web 빌드가 `user://` 파일, `localStorage`, 단일 쿠키를 사용하지만 재접속 때
진행을 복원하지 못하는 사례가 있습니다.

- 저장 시각이 같은 초인 두 저장본은 더 최신 내용을 구분할 수 없습니다.
- `localStorage`는 한 칸뿐이라 쓰기 중 손상되면 이전 정상본이 없습니다.
- 저장 JSON이 단일 쿠키 제한을 넘으면 쿠키 복구본을 만들지 않습니다.
- 브라우저 백업 실패 여부가 사용자에게 구체적으로 보이지 않습니다.
- 브라우저 저장은 주소별로 분리되므로 다른 도메인으로 접속하면 자동 공유되지
  않습니다.

## 2. 저장 규칙

- 스키마 9에 `save_revision`을 추가합니다.
- 저장할 때마다 revision을 1 증가시킵니다.
- 불러올 때 revision을 먼저 비교하고, 같을 때만 Unix 저장 시각을 비교합니다.
- Web `localStorage`는 현재본과 직전 정상본을 각각 보관합니다.
- 쿠키는 URI 인코딩한 저장 문자열을 여러 조각으로 나눠 보관하고 다시 읽어
  검증합니다.
- 파일 현재본·파일 백업·Web 현재본·Web 백업·쿠키본 중 가장 최신 정상본을
  선택합니다.

## 3. 사용자 표시

- 불러온 출처를 저장 상태에 표시합니다.
- 수동 저장 뒤 Web 파일과 브라우저 백업 중 실제 성공한 경로를 안내합니다.
- 어느 경로도 지속 저장되지 않으면 `저장 제한`으로 표시합니다.

## 4. 호환성과 제외 범위

- 스키마 1~8 저장은 revision 0으로 이관한 뒤 다음 저장부터 증가합니다.
- 기존 `jjugumeong.save.v1` 값과 단일 `jjugumeong_save` 쿠키를 계속 읽습니다.
- 서로 다른 도메인 사이의 자동 저장 이전은 브라우저 보안 경계 때문에 이번
  범위에 포함하지 않습니다.
- 계정 서버·온라인 동기화는 추가하지 않습니다.

## 5. 완료 조건

- 같은 Unix 초의 저장본은 더 높은 revision이 선택됩니다.
- 손상된 Web 현재본이 있어도 Web 백업 또는 쿠키본을 선택합니다.
- 기존 스키마 1~8 저장을 잃지 않고 스키마 9로 이관합니다.
- 빈 저장과 저장 재실행에서 전체 Godot 테스트가 통과합니다.
- Web export, PWA 저장 셸 검증과 프로덕션 빌드가 통과합니다.
